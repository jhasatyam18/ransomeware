Failback Behaviour

Datamotive solution prescribes appropriate Entity Type and Replication Type depending on the state of system and infrastructure.

Making changes to the prescribed values may incur additional storage consumption and time required for replication. This in-turn may result into higher costs.

Following use cases describe resultant target infrastructure storage and virtual instances after performing "Reverse" operation.

Case 1: Entity Type: Maintain Original, Replication Type: Differential

TYPE PLATFORM INFRASTRUCTURE STATE
Target Instance All
  • Original Instance
Target Storage VMware
  • Original VM will get reverted to last replicated snapshot.
  • Any externally created snapshots after the last replicated snapshot will get deleted.
  • Differential data will be written back to same Virtual Disks.
AWS
  • Original EBS Volumes will get detached and deleted.
  • New snapshot will be created from last replicated snapshot.
  • Differential data will be written back to new snapshot.
Azure
  • Original managed disks will get detached and deleted.
  • New page blobs or managed disk (disks greater than 8TB) will be created from last replicated snapshot.
  • Differential data will be written back to new page blob or managed disk.
Impact All
  • None

Case 2: Entity Type: Maintain Original, Replication Type: Full

TYPE PLATFORM INFRASTRUCTURE STATE
Target Instance All
  • Original Instance
Target Storage VMware
  • Original disks will get detached and deleted.
  • New Virtual Disks will be created & attached to the instance.
  • Full data will be written back to new Virtual Disks.
AWS
  • Original EBS Volumes will get detached and deleted.
  • New replication snapshot will be created.
  • Full data will be written back to new snapshot.
Azure
  • Original managed disks will get detached and deleted.
  • New page blob or managed disk (disks greater than 8TB) will be created.
  • Full data will be written back to new page blob or managed disk.
Impact All
  • Full data will be used for egress charges.

Case 3: Entity Type: Create New Copy, Replication Type: Differential

TYPE PLATFORM INFRASTRUCTURE STATE
Target Instance VMware
  • New Instance with suffix "-dm-reversed" will get cloned from last replicated snapshot.
  • Original instance with its data will remain in system.

AWS,

Azure
  • New Instance with suffix "-dm-reversed" will get created.
  • Original instance with its data will remain in system.
Target Storage VMware
  • Differential data will be written back to newly cloned VM.
AWS
  • New replication snapshot will get created from last replicated snapshot.
  • Differential data will be written back to new snapshot.
Azure
  • New replication page blob or managed disk (disks greater than 8TB) will get created from last replicated snapshot.
  • New page blob or managed disk (disks greater than 8TB) will be created.
  • Differential data will be written back to new page blob or managed disk.
Impact All
  • Additional storage size equal to the provisioned storage size of the instance, compute resources and IPs will be required.

Case 4: Entity Type: Create New Copy, Replication Type: Full

TYPE PLATFORM INFRASTRUCTURE STATE
Target Instance All
  • A new VM with suffix "-dm-reversed" will get created.
  • Original VM with its data will remain in system.
Target Storage VMware
  • New Virtual Disks will be created & attached to the instance.
  • Full data will be written back to new Virtual Disks.
AWS
  • New replication snapshot will be created.
  • Full data will be written back to new snapshot.
Azure
  • New page blob or managed disk (disks greater than 8TB) will be created.
  • Full data will be written back to new page blob or managed disk.
Impact All
  • Additional storage size equal to the provisioned storage size of the instance, compute resources and IPs will be required. Full data will be used for egress charges.
